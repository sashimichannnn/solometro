## Chapter 1｜プロダクトの思想的土台

### 障害の社会モデル — solometroの立場

> **「移動できない」のは、情報が整っていない社会の側の問題だ。**

障害の社会モデルとは、障害を「個人の心身機能の問題」としてではなく、「社会の側にあるバリアが生み出すもの」として捉える考え方だ。

段差は物理的なバリアだが、**「その段差の情報にアクセスできないこと」もまたバリアである。**

solometroが取り除こうとしているのは段差そのものではなく、情報の分散・不透明さ・アクセスの困難さという、**社会が作り出した情報バリア**だ。

| | 医療モデル（採用しない） | 社会モデル（solometroの立場） |
|---|---|---|
| 問題の所在 | 個人の心身機能 | 社会の設計 |
| 見方 | 「車椅子だから移動できない」 | 「情報が整っていないから移動できない」 |
| 誰が変わるべきか | 個人が工夫・妥協すべき | 社会・ツールが変わるべき |
| 提供の姿勢 | 「設備を使わせてあげる」 | 「当然の移動の権利を支える」 |

**solometroは「社会モデルを実装するツール」として設計する。**

## Chapter 2｜設計原則 — 仕様に迷ったときの指標

### 原則 1｜情報へのアクセス自体をフラットにする

段差という物理バリアの前に、**「情報バリア」がある。** PDFの重さ、分散した情報源、専門用語、検索の手間——これらすべてが移動の障壁だ。solometroは情報をワンストップ・即時・平易に提供することで、情報バリアを取り除く。

**仕様判断の問い：「この機能は情報へのアクセスを楽にするか、複雑にするか？」**

- ✓ 採用する：1タップで確認できる / 読まなくてもわかるアイコン / 事前に全情報が揃っている
- ✗ 避ける：複数ページをたどる必要がある / 専門用語の多用 / 「詳細はPDFで」

---

### 原則 2｜「できない」ではなく「何をすればできるか」を示す

「この駅は単独乗降不可」で終わるのは医療モデルの発想だ。社会モデルに立つなら、**「代替ルートはこちら」「係員に〇〇を依頼する」** と、ユーザーが次の行動を選べる情報を返す。選択肢を渡すことが自律的な移動を支える。

**仕様判断の問い：「この表示はユーザーの次の行動につながるか？」**

- ✓ 採用する：代替ルートの提示 / 「この号車なら乗れる」と具体的に示す / 係員依頼の方法を明示
- ✗ 避ける：「要係員対応」だけで終わる / ✕マークのみ表示 / 理由なしの「利用不可」

---

### 原則 3｜健常者との体験格差を設計の基準にする

「健常者がYahoo!乗換で10秒で経路確定できるなら、solometroも同じ水準を目指す」——これがUXの基準線だ。**「車椅子ユーザーだからある程度手間がかかるのは仕方ない」という前提を設計に持ち込まない。**

**仕様判断の問い：「健常者なら何ステップで完了するか？その差は何ステップか？」**

- ✓ 採用する：ステップ差ゼロに近づける設計 / 1アクションで完結 / 事前準備なしで使える
- ✗ 避ける：「仕方ない」の妥協 / 健常者と別導線 / 「慣れれば使える」設計

---

### 原則 4｜不確かな情報は隠さず、判断をユーザーに渡す

情報の欠如や不確実性を隠すことは、別のバリアを生む。**「この座標は概算値です」「データは〇年〇月時点」** と明示することで、ユーザーが自分で判断できる。完璧なデータを待つより、**不完全であることを透明に示す方が誠実で安全だ。**

**仕様判断の問い：「この情報の信頼度と出典をユーザーに伝えているか？」**

- ✓ 採用する：データの出典と取得日を明示 / 推定値は明確にラベル / 「現地確認を推奨」を添える
- ✗ 避ける：出典不明データの断定表示 / 古いデータを最新のように見せる / 不確かさを隠して安心感を演出

## Chapter 3｜プロダクト定義

### ミッション

> **「駅員なしで目的地にたどり着ける」を、一本道で示す。**
>
> 出発地から目的地まで、どのルートで行き、どの号車に乗り、エレベーターはどこにあるかを——地上からも地下からも迷わず——健常者がYahoo!乗換を使うのと同じ手軽さで提示する。

---

### 解決する3つの課題

#### 課題① ルート｜駅員ゼロで行けるルートが自分でしかわからない

- **現状の痛点：** Yahoo!乗換でルート検索 → 各駅を個別にGoogle検索 → 公式PDFで確認 → 脳内で統合。乗換のたびに「ここ段差あって乗れるっけ？」を自分で判断している。知見がないと移動できない。
- **solometroが解決：** 全経路上の乗換駅の単独乗降可否＋乗車号車を一画面で提示する。

#### 課題② 地上→地下｜駅のエレベーター入口が地上から見つからない

- **現状の痛点：** GoogleマップはZoom inしないとエレベーターのピンが見えない設計。現地に着いてからどっちに歩けばいいか直感的にわからない。
- **solometroが解決：** 出口番号＋「〇〇交差点角、右に50m」レベルの文字情報＋座標ベースのマップ直リンクを提供する。

#### 課題③ 地下→地上｜降りた後に地上へのエレベーターが地下でも迷う

- **現状の痛点：** 降車後、案内板を頼りに地上行きエレベーターを探す。乗換動線上のエレベーター位置も把握できていない。
- **solometroが解決：** 「改札を出て右、約30m」レベルの地下動線テキスト＋バリアフリー便利帳の経路情報を反映する。

---

### ユーザーが体験する一本道のフロー

```
Yahoo!乗換 or Googleマップでルートを調べる（従来通り）
  ↓
ブラウザの「共有」ボタン → solometroに渡す（Web Share Target）
  ↓
【駅員ゼロルート提示】乗換駅 × 号車  ← solometroの固有価値
  ↓
【EV入口確認】地上↔地下の双方向  ← solometroの固有価値
  ↓
Googleマップで現地ナビへシームレス移行
```

## Chapter 4｜機能要件

### バリアフリールート提示｜Must

出発駅・目的駅を指定すると、乗換を含む全経路の単独乗降可否を判定して表示。単独不可の乗換がある場合は代替ルートを提示、または要係員であることと依頼方法を明示する（原則2）。

### 乗車号車・ドア位置の明示｜Must

単独乗降できる号車番号とドア位置を路線×駅×方面で表示。乗換時に「何号車に乗ればエレベーターが近いか」まで示す。

### エレベーター入口ナビ（双方向）｜Must

**地上→地下（乗る前）**
- 出口番号＋目印テキスト
- 「〇〇交差点角、右50m」レベルの文字情報
- 座標ベースのGoogleマップ直リンク（ルート検索・ピン表示の2ボタン）

**地下→地上（降りた後）**
- 改札からの動線テキスト（「改札を出て右、約30m」）
- バリアフリー便利帳の経路情報を反映

### Yahoo!乗換 / Googleマップからの共有受け取り｜Must

Web Share Target APIによりブラウザの「共有」ボタンからsolometroを呼び出せる。
共有されたURLから出発駅・目的駅を自動解析し、バリアフリー情報をそのまま表示する。

- 対応ソース：Yahoo!乗換、Googleマップ（ルート検索URL）
- 実装ファイル：`manifest.json`（share_target定義）、`sw.js`、`share.html`
- 対応OS：Android（Chrome）優先、iOS（Safari 16.4以降）

### エレベーター運休情報｜実装済み

メトロ公式サイトをbot巡回してリアルタイム反映。ルート上に運休EVがある場合は警告表示（ルート提示との統合が次の課題）。

### AIアシスト「チエバ」｜API統合待ち

Gemini連携による自然言語サポート。本体機能が安定してから統合。現時点はスタンドアロンの補完機能。

## クロージング

> **solometroは、移動の自由を「特別な配慮」ではなく「当然の権利」として設計する。**
>
> バリアは社会が作った。だから社会のツールが取り除く。情報が整えば、人は動ける。solometroはその「情報を整える」部分を担う。
